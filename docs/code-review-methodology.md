# Code Review Methodology

Claude Code의 `/code-review` 커맨드를 활용한 코드 리뷰 방법론.

---

## 개요

7가지 각도로 후보를 발굴하고, 검증 단계에서 거짓 양성을 걸러낸 뒤, 확정된 건만 수정하는 3단계 프로세스.

```
Phase 1 — 후보 발굴 (7개 각도, 각 최대 6개)
    ↓
Phase 2 — 검증 (CONFIRMED / PLAUSIBLE / REFUTED)
    ↓
Phase 3 — 수정 (--fix) 또는 PR 코멘트 (--comment)
```

---

## Phase 1 — 후보 발굴

### 버그 각도 (정확성)

| 각도                | 목적                                 | 주요 체크 포인트                                                         |
| ------------------- | ------------------------------------ | ------------------------------------------------------------------------ |
| **A. 라인별 스캔**  | diff 전체를 한 줄씩 읽기             | 잘못된 조건, null 역참조, `await` 누락, 에러 삼킴, 변수 혼동, off-by-one |
| **B. 삭제된 동작**  | 제거된 코드가 보장하던 불변식 추적   | 삭제된 가드/검증/에러 경로가 새 코드에서 복원됐는지                      |
| **C. 파일 간 추적** | 변경된 함수의 호출부와 피호출부 확인 | 새 전제조건, 변경된 반환 형태, 인터페이스 불일치                         |

### 정리 각도 (품질)

| 각도          | 목적                                    | 주요 체크 포인트                                          |
| ------------- | --------------------------------------- | --------------------------------------------------------- |
| **D. 재사용** | 코드베이스에 이미 있는 기능 재구현 탐지 | 인접 파일, 공통 유틸 모듈에서 동일 기능 검색              |
| **E. 단순화** | 불필요한 복잡성 제거                    | 중복 상태, copy-paste, 깊은 중첩, 데드코드                |
| **F. 효율**   | 낭비되는 연산 제거                      | 반복 I/O, 순차 처리할 수 있는 병렬 작업, 핫 패스의 블로킹 |

### 설계 각도

| 각도                  | 목적                 | 주요 체크 포인트                                       |
| --------------------- | -------------------- | ------------------------------------------------------ |
| **G. 고도(Altitude)** | 밴드에이드 패치 탐지 | 근본 원인 수정 대신 특정 케이스만 처리하는 레이어 추가 |

---

## Phase 2 — 검증

각 후보에 대해 독립적인 검증자가 세 가지 중 하나를 판정한다.

| 판정          | 의미             | 기준                                         |
| ------------- | ---------------- | -------------------------------------------- |
| **CONFIRMED** | 확실한 버그      | 코드에서 직접 추적 가능                      |
| **PLAUSIBLE** | 발생 가능한 버그 | 현실적인 런타임 조건에서 발생 가능           |
| **REFUTED**   | 거짓 양성        | 코드상 명백히 불가능함을 증명할 수 있을 때만 |

**PLAUSIBLE 기본값 원칙** — 다음과 같은 조건은 PLAUSIBLE로 통과시킨다:

- 네트워크 중단, 타이밍 레이스
- 선택적 필드 누락, 빈 리스트, null 역참조
- 경계값 off-by-one
- 에러 핸들러, 콜드 캐시, 재시도 경로

CONFIRMED + PLAUSIBLE만 수정 대상에 포함한다.

---

## Phase 3 — 수정 방식

| 플래그      | 동작                             |
| ----------- | -------------------------------- |
| `--fix`     | 로컬 워킹트리에 직접 수정 적용   |
| `--comment` | GitHub PR에 인라인 코멘트로 게시 |
| (없음)      | JSON 리포트만 출력               |

---

## 우선순위

1. **버그 (정확성)** — 항상 정리보다 우선
2. **정리 (재사용/단순화/효율)**
3. **설계 (고도)**

최대 10개 findings 출력. 넘으면 심각도 순으로 상위 10개만 유지.

---

## 사용 예시

```bash
# 현재 브랜치 리뷰 후 로컬 수정
/code-review --fix

# PR 리뷰 후 GitHub 인라인 코멘트 게시
/code-review --comment

# PR 번호 지정
/code-review ultra 1
```
