# PRD — Source Display Frontend

## 1. 개요

AI 답변에 참고한 문서의 출처를 뱃지로 표시하고, 클릭 시 해당 청크 원문을 키워드 하이라이트와 함께 사이드패널에서 확인할 수 있는 기능.

---

## 2. 사용자 스토리

| ID    | 역할   | 스토리                                            | 완료 조건                                                 |
| ----- | ------ | ------------------------------------------------- | --------------------------------------------------------- |
| US-01 | 실무자 | 답변이 어떤 문서를 참고했는지 알고 싶다           | 답변 하단에 파일명(+페이지) 출처 뱃지가 최대 3개 표시된다 |
| US-02 | 실무자 | 출처 원문을 직접 확인해서 답변을 검증하고 싶다    | 뱃지 클릭 시 해당 청크 원문이 사이드패널에 표시된다       |
| US-03 | 실무자 | 원문에서 내 질문과 관련된 부분을 빠르게 찾고 싶다 | 사이드패널 원문에서 질문 키워드가 하이라이트된다          |
| US-04 | 실무자 | 사이드패널을 쉽게 닫고 싶다                       | 닫기 버튼 또는 바깥 영역 클릭으로 패널이 닫힌다           |

---

## 3. 기술 결정 (ADR)

### ADR-01: 패널 상태 관리 아키텍처

**Context** — 어떤 뱃지가 클릭됐는지, 사이드패널이 열려있는지를 어디서 관리할지 결정이 필요했다. MessageBubble 내부, useChat 확장, 별도 훅의 세 가지 방안을 검토했다.

**Decision** — `useSourcePanel` 훅을 신규 생성하여 패널 상태를 독립적으로 관리한다. ChatWindow가 두 훅(useChat, useSourcePanel)을 모두 호출한다.

**Alternatives**

- _MessageBubble 내부 (안 1)_ — 구현 단순하나 각 메시지가 독립적으로 패널을 가져 동시에 여러 패널이 열릴 수 있음. MessageBubble에 UI와 상태가 섞여 역할 과부하
- _useChat 확장 (안 2)_ — 한 곳에서 관리해 패널 하나만 열림을 보장하지만, 채팅 상태와 패널 상태가 한 훅에 섞여 useChat 책임이 비대해짐

**Consequences**

- (+) useDocuments, useChat과 동일한 훅 분리 패턴으로 코드베이스 일관성 유지
- (+) useChat을 수정하지 않아 기존 채팅 코드에 영향 없음
- (+) 패널이 한 곳에서 관리되어 동시에 하나만 열림 보장
- (+) Day 11 문서 비교 기능에서 useSourcePanel 재사용 또는 확장 용이
- (-) 파일이 하나 더 생김 (useSourcePanel.ts)

---

### ADR-02: 키워드 하이라이트 방식

**Context** — 사이드패널 원문에서 질문과 관련된 부분을 강조하기 위한 하이라이트 방식을 결정해야 했다.

**Decision** — 질문을 공백으로 분리하여 청크에서 일치하는 단어를 `<mark>` 태그로 감싸는 단순 문자열 매칭 방식을 사용한다.

**Alternatives**

- _형태소 분석 기반 (konlpy 등)_ — "처리하다"와 "처리" 같은 어간을 연결해 더 정확한 하이라이트가 가능하지만, 별도 한국어 NLP 라이브러리 설치 및 백엔드 연동 필요. 포트폴리오 Day 5 범위를 초과

**Consequences**

- (+) 구현 단순 — 30분 이내 구현 가능
- (+) 외부 라이브러리 불필요, 프론트에서 독립적으로 처리
- (+) 포트폴리오 데모에서 시각적 효과로 충분히 임팩트 있음
- (-) 한국어 형태소 변형 미처리 — "처리하다"를 "처리"로 검색하면 하이라이트 안 됨
- (-) 조사/어미가 붙은 단어 불일치 가능성 있음

---

### ADR-03: 같은 파일의 여러 청크 처리

**Context** — 동일 파일에서 여러 페이지가 매칭될 때 뱃지를 파일당 1개로 합칠지, 청크별로 각각 표시할지 결정이 필요했다.

**Decision** — 청크별로 각각 표시한다. 같은 파일이라도 페이지가 다르면 별도 뱃지로 노출한다.

**Alternatives**

- _파일당 1개_ — UI가 단순하나 같은 파일에서 여러 관련 내용이 있을 때 정보 손실. 사용자가 다른 페이지의 내용을 놓침

**Consequences**

- (+) RAG가 청크 단위로 동작함을 시각적으로 명확히 드러냄 — 포트폴리오 어필 포인트
- (+) 같은 파일의 다른 페이지 정보를 모두 탐색 가능
- (-) 같은 파일 뱃지가 여러 개 나올 수 있어 반복적으로 보일 수 있음 (최대 3개 제한으로 완화)

---

## 4. Out of Scope

- PDF 원문 뷰어 (전체 페이지 렌더링)
- 출처 문서 다운로드
- 출처 3개 초과 표시 (페이지네이션 등)
- 형태소 분석 기반 키워드 하이라이트 (단순 문자열 매칭만 구현)
- 출처 뱃지 즐겨찾기/저장

---

## 5. 용어 정의

| 용어              | 정의                                                                |
| ----------------- | ------------------------------------------------------------------- |
| 출처 뱃지         | 답변 하단에 표시되는 참고 문서 정보 컴포넌트 (파일명 + 페이지 번호) |
| 사이드패널        | 우측에서 슬라이드인 방식으로 나타나는 청크 원문 확인 영역           |
| 청크              | 문서를 검색에 적합한 크기로 잘라낸 텍스트 단위                      |
| 키워드 하이라이트 | 질문의 단어를 청크 원문에서 찾아 시각적으로 강조하는 기능           |
