# PRD — Chat Backend

## 1. 개요

사용자 질문을 임베딩하여 ChromaDB에서 관련 청크를 검색하고, 이전 대화 history와 함께 Groq LLM에 전달하여 답변을 SSE 스트리밍으로 반환한다.

---

## 2. 사용자 스토리

| ID    | 역할   | 스토리                                              | 완료 조건                                                             |
| ----- | ------ | --------------------------------------------------- | --------------------------------------------------------------------- |
| US-01 | 실무자 | 질문과 관련된 문서 내용을 기반으로 답변을 받고 싶다 | 질문과 유사한 문서 청크가 검색되어 Groq 답변에 반영된다               |
| US-02 | 실무자 | 답변이 어떤 문서를 참고했는지 알고 싶다             | 검색에 사용된 청크의 출처 파일명이 sources 이벤트로 반환된다          |
| US-03 | 실무자 | 이전 대화 맥락으로 후속 질문을 하고 싶다            | 전달받은 history가 Groq messages에 포함되어 맥락 있는 답변이 생성된다 |

---

## 3. 기술 결정 (ADR)

### ADR-01: history 전달 방식

**Context** — 멀티턴 대화를 위해 이전 대화를 Groq에 어떻게 전달할지 결정이 필요했다.

**Decision** — `ChatRequest`에 `history` 필드를 추가하고, Groq messages 배열에서 system prompt와 현재 질문 사이에 삽입한다.

**Alternatives**

- _history 미전달_ — 구현 변경 불필요하지만 후속 질문이 맥락 없이 동작해 챗봇 사용성 저하
- _서버에서 세션 관리_ — 서버가 대화를 기억하는 방식이지만 세션 저장소 구현 필요. 포트폴리오 범위 밖

**Consequences**

- (+) 멀티턴 대화 지원으로 후속 질문이 맥락 있게 처리됨
- (+) `history` 기본값 `[]`로 기존 단일 턴 호출과 하위 호환
- (-) history가 길수록 Groq 토큰 비용 증가
- (-) messages 배열이 커질수록 Groq 응답 지연 가능성 있음

---

### ADR-02: Top K 청크 개수

**Context** — ChromaDB 검색 시 몇 개의 청크를 Groq context로 전달할지 결정이 필요했다. 개수가 많을수록 답변 품질은 높아지지만 토큰 비용과 응답 속도에 영향을 준다.

**Decision** — Top 5로 설정한다.

**Alternatives**

- _Top 3_ — 빠르고 저렴하지만 관련 내용이 누락될 가능성 있음
- _Top 10_ — context가 풍부하지만 토큰 비용 증가 및 응답 지연 발생 가능

**Consequences**

- (+) 품질과 속도의 균형점
- (+) 포트폴리오 데모에서 충분한 답변 품질 확보
- (-) 문서가 매우 많아질 경우 Top 5로는 관련 청크가 누락될 수 있음
- (-) 하드코딩 값이라 문서 규모에 따라 조정이 필요할 수 있음

---

## 4. Out of Scope

- 대화 기록 서버 저장 (history는 프론트에서 관리)
- 사용자 인증 / 세션 관리
- 문서 필터링 (특정 문서만 검색 범위 지정 — Day 11 기능)
- 스트리밍 중단 (답변 생성 취소)

---

## 5. 용어 정의

| 용어    | 정의                                                                              |
| ------- | --------------------------------------------------------------------------------- |
| 임베딩  | 텍스트를 벡터로 변환하여 의미 기반 유사도 검색이 가능하게 하는 과정               |
| Top K   | 유사도 검색에서 상위 K개의 청크를 선택하는 파라미터                               |
| history | 현재 질문 이전의 user/assistant 대화 목록. 프론트에서 전달받음                    |
| SSE     | Server-Sent Events — 서버가 클라이언트로 실시간 데이터를 스트리밍하는 단방향 방식 |
