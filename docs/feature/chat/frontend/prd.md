# PRD — Chat Frontend

## 1. 개요

업로드된 문서를 기반으로 AI에게 질문하고 답변을 받는 채팅 인터페이스. SSE 스트리밍으로 실시간 타이핑 효과를 제공하고, 이전 대화 맥락을 유지하는 멀티턴 대화를 지원한다.

---

## 2. 사용자 스토리

| ID    | 역할   | 스토리                                   | 완료 조건                                                             |
| ----- | ------ | ---------------------------------------- | --------------------------------------------------------------------- |
| US-01 | 실무자 | 문서에 대해 질문하고 싶다                | Enter 또는 전송 버튼으로 질문을 전송하면 답변이 스트리밍으로 표시된다 |
| US-02 | 실무자 | 답변이 어떤 문서를 참고했는지 알고 싶다  | 스트리밍 완료 후 답변 하단에 출처 뱃지가 표시된다                     |
| US-03 | 실무자 | 이전 대화 맥락으로 후속 질문을 하고 싶다 | 이전 대화를 참고한 답변이 반환된다                                    |
| US-04 | 실무자 | 긴 대화를 편하게 읽고 싶다               | 새 메시지가 추가될 때 자동으로 스크롤되고 user/assistant가 구분된다   |

## 3. 기술 결정 (ADR)

### ADR-01: 상태관리 아키텍처

**Context** — messages state와 sendMessage 로직을 어디에 둘지 결정이 필요했다. 채팅, 문서 업로드 외에도 기능이 추가될수록 page.tsx가 비대해질 수 있다.

**Decision** — `useChat` custom hook으로 상태와 로직을 분리한다. ChatWindow가 훅을 직접 호출한다.

**Alternatives**

- _Page-level 관리 (안 1)_ — 구현 단순하지만 page.tsx가 문서 + 채팅 state를 모두 관리해 비대해짐. useDocuments 패턴과도 불일치
- _Context API (안 3)_ — props drilling 없이 어디서든 접근 가능하지만 포트폴리오 규모에서 오버엔지니어링. 테스트 설정도 복잡

**Consequences**

- (+) useDocuments와 동일한 패턴으로 코드베이스 일관성 유지
- (+) 훅 단독 테스트 가능
- (+) page.tsx가 얇게 유지되어 기능 추가 시 복잡도 낮음
- (+) 나중에 Context 필요 시 useChat을 ChatProvider로 감싸는 방식으로 자연스럽게 마이그레이션 가능
- (-) 파일이 하나 더 생김 (useChat.ts)
- (-) 훅 내부 로직이 복잡해질수록 단일 파일이 비대해질 수 있음

---

### ADR-02: SSE 수신 방식

**Context** — `/chat`은 POST 엔드포인트라 question과 history를 body에 담아야 한다. SSE 수신 방식을 결정해야 했다.

**Decision** — `EventSource` 대신 `fetch` + `ReadableStream`으로 직접 SSE를 파싱한다.

**Alternatives**

- _EventSource_ — 브라우저 내장 SSE API로 사용이 간단하지만 GET 요청만 지원. history를 body에 담을 수 없어 사용 불가

**Consequences**

- (+) POST 요청으로 question + history를 body에 담아 전송 가능
- (+) EventSource보다 에러 처리 유연성이 높음 (응답 상태 코드 직접 확인 가능)
- (-) SSE 파싱 코드(`\n\n` 기준 이벤트 블록 분리)를 직접 구현해야 함
- (-) 버퍼 처리 등 네트워크 청크 경계를 직접 관리해야 함

---

### ADR-03: 대화 기록 유지 범위

**Context** — 대화 기록을 어디까지 유지할지 결정이 필요했다. 새로고침 후 복원이 필요한지 여부가 핵심이었다.

**Decision** — React state로만 관리한다. 탭을 닫거나 새로고침하면 대화가 사라진다.

**Alternatives**

- _localStorage_ — 새로고침해도 복원 가능하지만 직렬화/역직렬화 구현 필요. 이점 대비 비용이 큼
- _서버 저장_ — 여러 기기에서 접근 가능하지만 백엔드 DB + API 추가 필요. 포트폴리오 범위 밖

**Consequences**

- (+) 구현 단순, 별도 직렬화/역직렬화 로직 불필요
- (+) 민감한 대화 내용이 브라우저에 저장되지 않음
- (-) 새로고침하면 대화가 사라짐
- (-) 페이지 이동 후 돌아오면 대화 기록이 없어짐

---

### ADR-04: history 백엔드 전달 여부

**Context** — 이전 대화를 화면에만 표시할지, Groq API에도 전달할지 결정이 필요했다.

**Decision** — history를 백엔드에 전달하여 Groq가 이전 대화 맥락을 인식하게 한다.

**Alternatives**

- _프론트 표시용만 (옵션 A)_ — 백엔드 변경 불필요하지만 "방금 말한 그 API는?" 같은 후속 질문이 맥락 없이 동작해 사용성 저하

**Consequences**

- (+) 멀티턴 대화 지원으로 실사용성 향상
- (+) 후속 질문("그 부분 더 자세히 설명해줘")이 자연스럽게 동작
- (-) 대화가 길어질수록 Groq 토큰 비용 증가
- (-) history 전달로 인해 요청 payload 크기가 커짐

---

## 4. Out of Scope

- 대화 기록 영속성 (새로고침 후 복원 — localStorage/서버 저장)
- 대화 내보내기 / 저장
- 여러 대화방 (멀티 세션)
- 특정 문서만 선택해서 질문 (Day 11 기능)
- 음성 입력

## 5. 용어 정의

| 용어     | 정의                                                                              |
| -------- | --------------------------------------------------------------------------------- |
| SSE      | Server-Sent Events — 서버가 클라이언트로 실시간 데이터를 스트리밍하는 단방향 방식 |
| 스트리밍 | 응답 전체를 기다리지 않고 청크 단위로 실시간 렌더링하는 방식                      |
| 멀티턴   | 이전 대화 맥락을 유지하며 이어지는 다회 대화                                      |
| history  | 현재 질문 이전의 user/assistant 대화 목록                                         |
