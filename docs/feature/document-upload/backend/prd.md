# PRD — Document Upload Backend

## 1. 개요

사용자가 업로드한 `.pdf`, `.md`, `.txt` 파일을 텍스트로 변환하고 ChromaDB에 벡터로 저장해 AI 채팅에서 검색 가능한 상태로 만드는 백엔드 기능.

---

## 2. 사용자 스토리

| ID    | 역할   | 스토리                                                | 완료 조건                                                                |
| ----- | ------ | ----------------------------------------------------- | ------------------------------------------------------------------------ |
| US-01 | 담당자 | 파일을 업로드하면 바로 응답을 받고 싶다               | 업로드 즉시 `indexing` 상태와 문서 ID를 반환한다                         |
| US-02 | 담당자 | 파일 처리가 완료되면 알고 싶다                        | SSE로 `ready` 상태 변경이 프론트에 전달된다                              |
| US-03 | 담당자 | 문서를 삭제하고 싶다                                  | `DELETE /docs/:id` 호출 시 ChromaDB에서 해당 문서 청크가 삭제된다        |
| US-04 | 실무자 | 현재 질문 가능한 문서 목록을 보고 싶다                | `GET /docs` 호출 시 문서 목록(`id`, `name`, `size`, `status`)이 반환된다 |
| US-05 | 담당자 | 같은 파일을 다시 올리면 최신 내용으로 갱신되길 원한다 | 파일명 기반 ID로 기존 문서를 덮어쓴다                                    |

---

## 3. 기술 결정 (ADR)

### ADR-01: 동기 vs 비동기 처리

**Context** — 청킹 + 임베딩 + ChromaDB 저장까지 수십 초가 걸릴 수 있다. 이 작업이 끝날 때까지 API 응답을 기다리면 프론트가 블로킹된다.

**Decision** — 업로드 즉시 `indexing` 상태를 반환하고, 처리는 백그라운드에서 수행한다.

**Alternatives**

- _동기 처리_ — 구현 단순하지만 큰 파일일수록 응답이 느리고 프론트 UX가 나빠짐

**Consequences**

- (+) 업로드 즉시 응답, 프론트 `indexing` 상태 UI와 자연스럽게 연결
- (+) 여러 파일 동시 업로드 시 병렬 처리 가능
- (+) 대용량 파일도 HTTP 타임아웃 없이 처리 가능
- (-) 백그라운드 작업 관리 필요, 처리 실패 시 SSE로 별도 에러 전달 필요
- (-) FastAPI `BackgroundTasks`는 서버 재시작 시 진행 중 작업이 사라짐 (작업 큐가 아님)
- (-) 동시 업로드가 많을 경우 임베딩 처리가 겹쳐 서버 리소스 과부하 가능

---

### ADR-02: 임베딩 모델 선택

**Context** — 텍스트를 벡터로 변환하는 임베딩 모델이 필요하다. OpenAI API는 유료다.

**Decision** — `sentence-transformers` 로컬 모델 사용.

**Alternatives**

- _OpenAI `text-embedding-3-small`_ — 품질 우수하지만 유료, API 키 필요

**Consequences**

- (+) 완전 무료, 외부 API 의존 없음
- (+) 로컬 실행이라 네트워크 레이턴시 없음
- (+) API 요금 걱정 없이 개발/테스트 가능
- (-) OpenAI 대비 임베딩 품질 낮을 수 있음, 특히 한국어 문서
- (-) 초기 모델 다운로드 필요 (수백 MB ~ GB), 배포 이미지 크기 증가
- (-) CPU만으로는 대용량 문서 임베딩 시 속도 저하

---

### ADR-03: LLM 선택

**Context** — 답변 생성에 LLM이 필요하다. OpenAI API는 유료다.

**Decision** — Groq API 사용 (무료 티어, Llama/Gemma 등 오픈소스 모델 제공).

**Alternatives**

- _OpenAI GPT_ — 품질 우수하지만 유료
- _Ollama 로컬 실행_ — 완전 무료지만 GPU 없으면 느림

**Consequences**

- (+) 무료 티어로 개발/포트폴리오 운영 가능
- (+) LPU 특성으로 OpenAI보다 토큰 생성 속도 빠름
- (+) Llama 3, Gemma 등 최신 오픈소스 모델 선택 가능
- (-) 무료 티어 제한 (분당 30 요청, 하루 14,400 요청)
- (-) 외부 API 의존이라 Groq 서비스 장애 시 영향받음
- (-) GPT-4 수준의 복잡한 추론은 기대하기 어려움

---

### ADR-04: 문서 ID 생성 방식

**Context** — 같은 파일을 재업로드 시 덮어쓰기를 하려면 "같은 문서"를 식별할 기준이 필요하다.

**Decision** — 파일명 기반으로 ID 생성. 같은 파일명 업로드 시 기존 문서 자동 덮어쓰기.

**Alternatives**

- _UUID_ — 매번 새 ID 생성. 같은 파일 재업로드 시 중복 청크 쌓임
- _파일 해시_ — 내용 기준 중복 감지 가능하지만 구현 복잡

**Consequences**

- (+) 구현 단순, 프론트의 "중복 파일명 처리는 백엔드에 위임" 결정과 일치
- (+) ChromaDB에서 ID로 바로 조회/삭제 가능, `GET /docs` 응답에서 ID 예측 가능
- (-) 파일명이 같지만 내용이 다른 파일을 올리면 의도치 않게 덮어써짐
- (-) 파일명에 특수문자, 공백 포함 시 ID 정규화 처리 필요
- (-) 동시에 같은 파일명으로 업로드 시 레이스 컨디션 가능성

---

### ADR-05: SSE 구독 방식

**Context** — 백그라운드 처리 완료 후 상태 변경을 프론트에 전달해야 한다.

**Decision** — 단일 스트림 (`GET /docs/status`)으로 모든 문서 상태 변경 이벤트 전송.

**Alternatives**

- _문서별 스트림 (`GET /docs/:id/status`)_ — 파일 N개 동시 업로드 시 N개 연결 필요
- _폴링_ — 구현 단순하지만 불필요한 요청 발생

**Consequences**

- (+) 파일 N개 동시 업로드해도 연결 수 1개로 고정
- (+) 채팅 스트리밍과 SSE 패턴 통일
- (+) 연결 끊김 시 재연결만으로 전체 상태 복구 가능
- (-) 클라이언트에서 문서 ID로 이벤트 필터링 로직 필요
- (-) 연결 유지 동안 서버 리소스 점유
- (-) 이벤트 유실 방지를 위한 `Last-Event-ID` 처리 고려 필요

---

### ADR-06: FastAPI 프레임워크 선택

**Context** — RAG 구현에 필요한 핵심 라이브러리(`sentence-transformers`, `pdfplumber`, `langchain`, `chromadb`)가 모두 Python 생태계에 있다. 백엔드 프레임워크를 선택해야 했다.

**Decision** — FastAPI 사용.

**Alternatives**

- _Flask_ — 단순하지만 async 지원이 약하고 SSE 구현이 번거로움
- _Node.js (Express/Hono)_ — 프론트와 언어 통일되지만 Python 라이브러리를 직접 사용 불가

**Consequences**

- (+) Python RAG 라이브러리(`sentence-transformers`, `pdfplumber`, `langchain`, `chromadb`) 그대로 사용
- (+) async/await 기본 지원으로 SSE 스트리밍 구현 용이
- (+) Pydantic으로 요청/응답 스키마 자동 검증, OpenAPI 스펙 자동 생성
- (+) 의존성 주입 시스템으로 테스트 시 모킹 편리
- (-) 프론트(TypeScript)와 언어가 달라 모노레포에서 타입 직접 공유 불가
- (-) Python GIL로 인해 CPU 집약적 작업(임베딩) 스레드 병렬화 제한
- (-) Node.js 대비 콜드 스타트 느리고 메모리 사용량 높음

---

### ADR-07: FastAPI 앱 구조

**Context** — 단일 파일로 시작하면 빠르지만 기능이 늘어날수록 관리가 어렵다. 어느 수준으로 구조화할지 결정이 필요했다.

**Decision** — `APIRouter` 기반 모듈화. 기능별 라우터 파일 분리, 비즈니스 로직은 라우터 안에.

```
backend/
├── main.py
├── routers/
│   ├── upload.py
│   └── docs.py
└── db/
    └── chroma.py
```

**Alternatives**

- _단일 파일 (main.py)_ — 구현 빠르지만 기능 추가 시 비대해짐
- _레이어드 아키텍처 (라우터/서비스/레포지토리)_ — 테스트 용이하지만 포트폴리오 규모에서 오버엔지니어링

**Consequences**

- (+) FastAPI 표준 방식, 레퍼런스 풍부
- (+) 기능 추가 시 라우터 파일만 추가, 기존 코드 수정 최소화
- (+) 라우터 파일 단위로 Git 충돌 줄어듦
- (+) 레이어드 아키텍처 (라우터/서비스/레포지토리 분리)로 마이그레이션 용이
- (-) 비즈니스 로직이 라우터에 섞여 완전한 단위 테스트 분리 어려움
- (-) 서비스 레이어 없이 직접 ChromaDB 호출 → 테스트 시 DB 모킹 어려움
- (-) 라우터가 두꺼워지면 결국 단일 파일과 비슷한 문제 발생 가능

---

## 4. Out of Scope

- OCR (이미지 기반 PDF 처리)
- 파일 디스크 저장 및 원본 다운로드
- 청크 크기 사용자 설정
- 문서 수정 / 내용 편집

---

## 5. 용어 정의

| 용어     | 정의                                                                                     |
| -------- | ---------------------------------------------------------------------------------------- |
| 청킹     | 문서를 검색에 적합한 작은 단위(청크)로 분할하는 과정                                     |
| 임베딩   | 텍스트를 숫자 벡터로 변환하는 과정. 의미적으로 유사한 텍스트는 벡터 공간에서 가깝게 위치 |
| ChromaDB | 벡터를 저장하고 유사도 검색을 수행하는 벡터 데이터베이스                                 |
| SSE      | Server-Sent Events — 서버가 클라이언트에 단방향으로 실시간 이벤트를 푸시하는 방식        |
| overlap  | 청킹 시 인접 청크 간 겹치는 토큰 수. 문맥 단절 방지용                                    |
