# PRD — Document Upload

## 1. 개요

PDF, MD, TXT 파일을 드래그앤드롭 또는 클릭으로 업로드하면 자동으로 인덱싱되어 AI 채팅에서 질문할 수 있는 상태로 준비되는 기능.

---

## 2. 사용자 스토리

| ID    | 역할   | 스토리                                                         | 완료 조건                                                             |
| ----- | ------ | -------------------------------------------------------------- | --------------------------------------------------------------------- |
| US-01 | 담당자 | PDF, MD, TXT 파일을 드래그앤드롭 또는 클릭으로 업로드하고 싶다 | 파일 선택 즉시 업로드가 시작된다                                      |
| US-02 | 담당자 | 여러 파일을 한 번에 올리고 싶다                                | 다중 파일 선택 후 동시에 업로드된다                                   |
| US-03 | 담당자 | 파일이 질문 가능한 상태인지 알고 싶다                          | 파일별 상태 뱃지(업로드 중 / 인덱싱 중 / 준비 완료 / 오류)가 표시된다 |
| US-04 | 담당자 | 오류가 난 파일을 쉽게 재시도하고 싶다                          | 오류 항목에 재시도 버튼이 표시되고, 클릭 시 재업로드가 시작된다       |

| US-06 | 담당자 | 페이지를 새로고침해도 문서 목록이 유지되길 원한다              | 페이지 진입 시 서버에서 문서 목록을 불러온다                          |

## 3. 기술 결정 (ADR)

### ADR-01: 중복 파일명 처리 방식

**Context** — 같은 이름의 파일을 두 번 업로드하려 할 때 프론트에서 막을지 결정이 필요했다. 단순히 파일명이 같아도 내용이 다른 파일일 수 있다 (예: 서로 다른 프로젝트의 `README.md`).

**Decision** — 프론트에서 차단하지 않고 백엔드(ChromaDB)에 위임한다.

**Alternatives**

- _차단_ — 구현 단순하지만 정상적인 케이스를 막아 사용자 경험을 해침
- _경고 후 허용_ — UX는 좋으나 불필요한 인터랙션 추가

**Consequences**

- (+) 파일명이 같아도 다른 내용의 문서를 자유롭게 업로드 가능
- (-) 완전히 동일한 파일을 중복 업로드해도 막지 못함. 벡터 DB에 중복 청크가 쌓일 수 있어 RAG 품질에 영향 가능

---

### ADR-02: 업로드 상태 관리 위치

**Context** — 파일 업로드 후 인덱싱까지 시간이 걸리는데, 여러 사용자가 같은 문서 풀을 공유하므로 상태를 어디서 관리할지 결정이 필요했다.

**Decision** — `uploading`은 프론트 로컬 state, `indexing` / `ready` / `error`는 백엔드 응답 기준으로 관리한다.

**Alternatives**

- _전부 백엔드_ — `uploading`도 서버에서 관리하면 새로고침 시 완전 복원 가능하지만, 파일 전송 중 상태를 실시간으로 서버에 반영하는 구현이 복잡함
- _전부 프론트_ — 새로고침하면 다른 사용자가 올린 문서 상태를 알 수 없어 공유 문서 풀 정책에 위배됨

**Consequences**

- (+) 구현 단순, 공유 상태는 서버 진실을 따름
- (-) `uploading` 중 새로고침하면 해당 상태가 사라짐. 서버 처리 결과가 `GET /docs`에 반영되기까지 짧은 공백 존재

---

### ADR-03: 로그인 없는 공유 문서 풀 채택

**Context** — 담당자(업로드)와 실무자(질문)가 다를 수 있어 문서 풀 공유가 필요했다. 동시에 로그인 구현은 포트폴리오 범위에서 복잡도가 크다.

**Decision** — 로그인 없이 모든 사용자가 동일한 문서 풀을 공유한다. 단, `uploadedBy?: string` 필드를 optional로 보유해 향후 확장에 대비한다.

**Alternatives**

- _로그인 도입_ — 사용자별 문서 격리 가능하지만 인증 시스템 구현 비용이 포트폴리오 범위를 벗어남
- _localStorage_ — 새로고침 유지는 되지만 다른 브라우저/사용자와 공유 불가

**Consequences**

- (+) 구현 단순, 팀 공유 시나리오에 자연스럽게 부합
- (-) 누가 올린 문서인지 추적 불가. 악의적 사용자가 문서를 무제한 업로드하거나 삭제할 수 있음

---

### ADR-04: 재시도 메커니즘 (File 객체 인메모리 보관)

**Context** — 오류 상태인 파일을 재업로드하려면 원본 File 객체가 필요하다. 백엔드가 없는 현재 단계에서 File 객체를 다시 얻을 방법을 결정해야 했다.

**Decision** — `UploadedDocument.file?: File` 필드에 File 객체를 인메모리로 보관한다. 재시도 버튼 클릭 시 이 객체를 재사용한다.

**Alternatives**

- _재시도 시 파일 탐색기 재오픈_ — 구현 단순하지만 사용자가 같은 파일을 직접 다시 찾아야 해서 UX 마찰이 큼
- _백엔드에서 재처리 트리거_ — 가장 이상적이지만 백엔드 구현 후에나 가능

**Consequences**

- (+) 재시도 UX가 자연스럽고 사용자 개입 불필요
- (-) 파일이 클수록 메모리 부담. 백엔드 연결 후에는 File 객체 보관이 불필요해지므로 제거 필요

---

### ADR-05: 상태관리 아키텍처 (Custom Hook)

**Context** — 문서 업로드 로직(`handleUpload`, `handleRetry`, `simulateUpload`)과 state를 어디에 둘지 결정이 필요했다. 채팅·출처 패널 등 기능이 추가되면 `page.tsx`가 비대해질 수 있다.

**Decision** — `useDocuments` custom hook으로 로직과 state를 분리한다. `page.tsx`는 hook을 호출하고 결과를 props로 전달하는 역할만 한다.

**Alternatives**

- _Page-level 관리 (안 1)_ — 구조 변경 없이 바로 구현 가능하지만 기능 추가 시 page.tsx가 비대해지고 단위 테스트가 어려움
- _Context API (안 3)_ — props drilling 없이 어디서든 접근 가능하지만 포트폴리오 규모에서 오버엔지니어링. 컴포넌트 재사용성도 떨어짐

**Consequences**

- (+) 로직 단위 테스트 가능, page.tsx 얇아짐, 향후 Context로 마이그레이션 용이
- (-) 파일이 하나 더 생김 (`useDocuments.ts`)

---

## 4. Out of Scope

- 로그인 / 인증 (추후 `uploadedBy` 필드로 확장 가능)
- 파일 수정 / 편집
- 문서 버전 관리
- 다중 선택 삭제
- 파일 미리보기

---

## 5. 용어 정의

| 용어         | 정의                                                                                          |
| ------------ | --------------------------------------------------------------------------------------------- |
| 인덱싱       | 문서를 청킹하고 벡터 임베딩하여 ChromaDB에 저장하는 과정                                      |
| 청킹         | 문서를 검색에 적합한 작은 단위로 나누는 과정                                                  |
| 공유 문서 풀 | 로그인 없이 모든 사용자가 동일한 문서 목록을 공유하는 구조                                    |
| RAG          | Retrieval-Augmented Generation — 문서 검색 결과를 컨텍스트로 활용해 AI가 답변을 생성하는 방식 |
| 준비 완료    | 인덱싱이 끝나 AI 채팅에서 질문 가능한 상태                                                    |
